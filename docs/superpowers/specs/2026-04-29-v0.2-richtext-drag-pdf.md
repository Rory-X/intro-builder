# intro-builder v0.2 — Rich Text + Drag Reorder + Puppeteer PDF

**Date:** 2026-04-29
**Status:** Draft, awaiting user review
**Depends on:** v0.1.1 (tagged, deployed)

## 1. What this solves

v0.1.1 shipped the usable shell; v0.2 raises the editor to production-grade:

1. **Bullet points in experience/projects/education are plain text**, losing all formatting. Replace with TipTap rich text — users can bold, italic, link, bulleted/numbered list, exactly like a real resume editor.
2. **Sections and items can't be reordered.** Users expect to drag "工作经历" above "教育背景," and drag their most recent job above an older one. Add section-level + item-level dnd-kit drag.
3. **PDF export is slow and doesn't match preview 1:1.** Replace @react-pdf/renderer (separate render tree) with puppeteer-chromium (screenshot the actual preview page) — pixel-identical, no font embedding.
4. **Old `string[]` data** in existing resumes needs transparent upgrade.

**Success criteria.**
1. User opens an experience entry, sees a TipTap toolbar (bold/italic/link/list), types rich text, autosave stores TipTap JSON in jsonb.
2. User grabs the drag handle on "项目经历" section and moves it above "工作经历" — preview instantly reorders.
3. User grabs the drag handle on a specific work entry and moves it above another entry within the same section.
4. "下载 PDF" returns an A4 PDF that is pixel-identical to the preview page (rendered by Chromium) in <6 seconds on Vercel Hobby.
5. User who had old `string[]` bullets opens their resume — see rich-text content (each old string becomes a paragraph), no data loss.

**Non-goals.** AI bullet rewrite (v0.3), collaborative editing, custom template upload, additional templates beyond Classic/Modern.

## 2. Data model changes

### 2.1 `ResumeContent` Zod schema — v2

```
Current (v1):
  experience[].bullets: string[]        // plain strings
  projects[].bullets: string[]
  education[].highlights: string[]

After (v2):
  experience[].content: TipTapJSON      // TipTap document JSON
  projects[].content: TipTapJSON
  education[].highlights: TipTapJSON    // same rich-text format
  sectionOrder: string[]                // ["basics","experience","education","projects","skills"]
```

`TipTapJSON` is `z.any()` at the Zod level (TipTap's JSON schema is complex; runtime validation of its internal shape is not worth the maintenance burden). We trust TipTap to output valid JSON and store/retrieve it faithfully.

Actually — more precise: we define:

```ts
const TipTapJSON = z.object({
  type: z.literal("doc"),
  content: z.array(z.any()).default([]),
}).passthrough();
```

This validates that the top-level has `type: "doc"` and `content` array, while being permissive about the inner node shapes. Good enough.

**`sectionOrder`** is a new top-level field on `ResumeContent` (not a DB column):

```ts
sectionOrder: z.array(z.string()).default(["basics","experience","education","projects","skills","custom"]),
```

Default covers all section keys. Editor and templates both read this array to determine render order. Missing section keys are appended at the end (forward-compatible).

### 2.2 Lazy migration strategy (frontend)

When `editor-client.tsx` receives `initialContent`:
1. Check if `experience[0].bullets` exists (old v1 format).
2. If yes → run `migrateV1toV2(content)` that converts each `bullets: string[]` to `content: TipTapJSON` by wrapping each string in a `<p>` node (or `<li>` inside `<ul>` — whatever TipTap's standard JSON representation is for a bulleted list item).
3. Also: add `sectionOrder` with default order if missing.
4. Immediately trigger a save with the migrated content.
5. The `ResumeContent` zod schema at v2 must accept BOTH shapes during parse (`safeParse`), or we do the migration BEFORE parse.

Decision: **migrate before parse.** The migration function runs on raw `unknown` data, produces v2 shape, then `ResumeContent.parse()` validates v2 only. This avoids polluting the schema with v1 compatibility unions.

The migration function lives at `lib/migrate-content.ts`:

```ts
export function migrateContent(raw: unknown): ResumeContent { ... }
```

Called in `editor-client.tsx` and in `app/(app)/resume/[id]/edit/page.tsx` (the RSC that fetches the row) — whichever is more appropriate. Putting it in the RSC means the client always gets v2 shape. That's cleaner.

Actually — the preview page at `/resume/[id]/preview` and the PDF route also need v2 data. So **migration runs in every code path that reads `row.content`**. Best to centralize: `db/helpers.ts` exports `getResumeContentV2(row): ResumeContent` that does parse + optional migration. All consumers call this instead of raw `row.content`.

### 2.3 Database schema

No Drizzle schema change needed. `content` is already `jsonb` — the structure change lives entirely inside the JSON. No SQL migration required.

`sectionOrder` lives inside `content` (not a separate column) to keep jsonb as the single source of truth for the resume body.

## 3. TipTap integration

### 3.1 Packages

```bash
pnpm add @tiptap/react @tiptap/starter-kit @tiptap/extension-link @tiptap/extension-underline @tiptap/pm
```

`@tiptap/starter-kit` includes: Document, Paragraph, Text, Bold, Italic, Strike, Code, Heading, BulletList, OrderedList, ListItem, Blockquote, HorizontalRule, HardBreak, History.

We'll use a minimal subset for resume bullets: **Bold, Italic, Underline, Link, BulletList, OrderedList, ListItem, Paragraph**.

No headings (section titles are structural, not user-typed). No images. No code blocks.

### 3.2 Component: `RichTextEditor`

`components/editor/rich-text-editor.tsx` — a reusable `"use client"` component:

Props: `{ content: TipTapJSON; onChange: (json: TipTapJSON) => void; placeholder?: string }`

Renders:
1. A floating toolbar with: Bold | Italic | Underline | Link | BulletList | OrderedList
2. The TipTap `<EditorContent>` area
3. On every update (`onUpdate`), calls `onChange` with `editor.getJSON()`

Styling: minimal border, matches shadcn Input aesthetic. Toolbar icons from lucide-react.

### 3.3 Replacing textarea in section editors

`experience-editor.tsx` currently renders a `<Textarea>` for bullets with `value={bullets.join("\n")}` / `onChange → split("\n")`. Replace with:

```tsx
<RichTextEditor
  content={field.content}
  onChange={(json) => setValue(`experience.${idx}.content`, json, { shouldDirty: true })}
  placeholder="描述你的工作成果…"
/>
```

Same for `projects-editor.tsx` and `education-editor.tsx` (highlights).

### 3.4 Rendering TipTap JSON in templates

Templates need to render the rich-text JSON to React DOM (Layout.tsx) and to a printable page (for puppeteer).

Since puppeteer screenshots the preview page (which uses Layout.tsx), we only need ONE renderer — the DOM one. The old `Pdf.tsx` files (react-pdf) are **deleted entirely** in v0.2.

For Layout.tsx, we need a `RichTextRenderer` component:

`components/preview/rich-text-renderer.tsx`:

```tsx
// Server-safe (no hooks) — renders TipTap JSON to static HTML
export function RichTextRenderer({ content }: { content: TipTapJSON }) {
  return <div dangerouslySetInnerHTML={{ __html: generateHTML(content, extensions) }} />;
}
```

Using `@tiptap/html` package's `generateHTML()` for server-safe rendering without instantiating an editor. This outputs plain HTML string from the JSON. We wrap it in a styled `<div>`.

Actually — `generateHTML` requires the same extensions list (StarterKit etc.). Add:

```bash
pnpm add @tiptap/html
```

## 4. Drag-and-drop (section + item level)

### 4.1 Section-level reorder

The editor renders sections in `sectionOrder` order. Each section wrapper gets a drag handle (grip icon) + `useSortable` from `@dnd-kit/sortable`.

`components/editor/section-sortable.tsx`:

Wraps each section editor in a `<SortableItem>` that:
- Shows a horizontal grip bar at the top of each section
- On drag-end, updates `sectionOrder` in form state

The `DndContext` for sections lives in `editor-client.tsx`, wrapping all section editors.

### 4.2 Item-level reorder

Within each array section (experience, education, projects, skills), each item gets a smaller vertical drag handle. Uses a **nested** `DndContext` inside each section editor.

`@dnd-kit/sortable` supports nested contexts natively — the inner context handles item-level drags, the outer context handles section-level drags.

On drag-end inside a section:
- `arrayMove` from `@dnd-kit/sortable` reorders the `useFieldArray` fields
- `move()` from `useFieldArray` performs the actual form-state reorder

### 4.3 Preview reorder

Both `ClassicLayout` and `ModernLayout` currently hardcode the section render order. Change them to:
1. Accept `sectionOrder: string[]` prop
2. Iterate `sectionOrder` and render each section via a switch/map
3. If a section key is unknown, skip it silently (forward-compatible)

## 5. PDF via puppeteer-chromium

### 5.1 Architecture

```
GET /api/pdf/[id]
  → auth check
  → fetch resume from DB
  → launch Chromium (via @sparticuz/chromium)
  → navigate to /resume/[id]/preview?_pdf=1 (internal URL, same Vercel deployment)
  → wait for fonts + layout
  → page.pdf({ format: 'A4', printBackground: true })
  → return PDF stream with Content-Disposition: attachment
```

`/resume/[id]/preview` already exists conceptually (the preview panel). We need to create an actual route that renders the template full-page, **styled exactly like print**: A4 aspect, white background, correct fonts.

New route: `app/(app)/resume/[id]/preview/page.tsx`

This page:
- Fetches the resume
- Runs migration
- Renders `<ClassicLayout>` or `<ModernLayout>` full-page
- Adds `<style>` for `@page { size: A4; margin: 0; }` and `body { -webkit-print-color-adjust: exact }`
- When `?_pdf=1` query param present, does NOT render the AppShell header (cleaner PDF)

### 5.2 Packages

```bash
pnpm add @sparticuz/chromium puppeteer-core
pnpm add -D @types/puppeteer-core
```

### 5.3 Performance budget

Vercel Hobby: **10-second function timeout**, **256 MB memory** (or 1024 MB with streaming — check).

Chromium cold start: ~2-3s. Page render: ~0.5-1s. PDF generation: ~0.5s. Total: **~3-5s** typical, well within 10s.

Optimization levers if it gets tight:
- Pre-warm: `/api/pdf/warm` endpoint that launches + kills Chromium (called by a cron)
- Reduce Chromium size: `@sparticuz/chromium` is already stripped (~50 MB)
- If still too slow: fall back to `window.print()` client-side as the primary path, offer server PDF as "beta"

### 5.4 Cleanup

Delete:
- `lib/templates/classic/Pdf.tsx`
- `lib/templates/modern/Pdf.tsx`
- `public/fonts/NotoSansSC-Regular.otf` (16 MB — not needed; Chromium uses system/web fonts)

Remove from `package.json`:
- `@react-pdf/renderer`

This shrinks the repo by ~16 MB and removes a complex dependency.

## 8. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| TipTap JSON too large for jsonb | Resume content is small (<50 KB even with rich text); not a concern at this scale |
| Nested DndContext conflicts | dnd-kit supports nested contexts; inner context handles items, outer handles sections; collision detection scoped per context |
| Chromium too slow on Vercel Hobby | Budget is 3-5s out of 10s; if tight, pre-warm via cron or fall back to client `window.print()` |
| `@sparticuz/chromium` layer size exceeds Vercel limit | Package is ~50 MB compressed; Vercel Hobby allows 250 MB uncompressed function size; should be fine |
| Lazy migration misidentifies v2 content as v1 | Check for `experience[0].content` (v2) vs `experience[0].bullets` (v1); if neither exists, treat as empty/new |
| TipTap update callback fires too frequently | Already have 2s debounce on autosave; TipTap `onUpdate` feeds into form state which feeds into the existing debounced save |
| `generateHTML` requires exact same extensions as editor | Centralize extensions in `lib/tiptap-extensions.ts`, import in both editor and renderer |

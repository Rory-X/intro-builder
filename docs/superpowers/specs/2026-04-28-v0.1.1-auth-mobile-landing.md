# intro-builder v0.1.1 — Auth UX + Mobile + Landing Design Spec

**Date:** 2026-04-28
**Status:** Draft, awaiting user review
**Depends on:** v0.1.0 (22 commits, merged)

## 1. Why this exists

v0.1.0 shipped the data & auth engine but the front-end is visually bare and lacks session/logout UX. Internet job-seekers landing on the site today see a H1 + 1 button; after login there's no way to sign out; the editor is unusable below `lg`. v0.1.1 closes these three gaps so the product is actually shippable to users.

**Success criteria.**
1. A visitor landing on `/` on either desktop or mobile understands what the product does within 5 seconds and sees a templatized preview.
2. A logged-in user sees their email in the top-right and can sign out in two clicks, everywhere.
3. A user on a 390px-wide phone can edit all five sections and toggle to preview, without any horizontal scroll or clipped controls.
4. Sessions auto-extend on each use (14-day sliding window) so users don't get logged out mid-editing after a few days.

**Non-goals.** Rich-text content, section reordering, PDF replacement — all deferred to v0.2.

## 2. Architecture changes

### 2.1 Global shell — new `<AppShell>` component

A single `<AppShell>` renders the `<Header>` (and nothing else) around every page except `/login`, `/verify-request`, and `/r/[slug]`.

**Mount strategy.** App Router's nested layouts:
- `app/layout.tsx` — stays minimal (html/body/Toaster). Does not render the header (keeps `/login` clean).
- `app/(marketing)/layout.tsx` — new group layout for `/` (landing). Renders `<AppShell>`.
- `app/(app)/layout.tsx` — already exists as a folder; promote to a real layout file that renders `<AppShell>` and wraps `/dashboard`, `/resume/[id]/edit`, `/resume/[id]/preview`.
- `/r/[slug]` keeps its bare wrapper (public, uncluttered).
- `/login`, `/verify-request` stay under `(auth)` group — no shell.

This gives us three public surfaces (marketing / auth / share) and one app surface, all distinguished in the URL tree, no conditional rendering inside shared components.

### 2.2 `<Header>` behavior

- **Not logged in** (observed via `await auth()` in the RSC): shows `intro-builder` brand on the left + "登录" link on the right. On mobile, just brand + login.
- **Logged in**: brand on the left, inline nav `[我的简历] [模板示例]` (hidden below `md`), avatar chip on the right (initials from email's local-part), dropdown with email + "我的简历" + "退出登录".
- **Mobile (< md)**: brand + hamburger icon opening a shadcn `<Sheet>` from the right containing the same items stacked vertically.

The header is a **Server Component** that performs `await auth()`. Interactive bits (dropdown, sheet) are small client islands inside it.

### 2.3 Logout — new Server Action

`app/(app)/actions/logout.ts` exports `signOutAction()`:
```ts
"use server";
import { signOut } from "@/lib/auth";
export async function signOutAction() {
  await signOut({ redirectTo: "/" });
}
```
Header's dropdown item is a `<form action={signOutAction}>`.

### 2.4 Session config — 14-day sliding window

Edit `lib/auth.ts` `session` block:
```ts
session: {
  strategy: "database",
  maxAge: 60 * 60 * 24 * 14,      // 14 days absolute
  updateAge: 60 * 60 * 24,        // extend session expiry at most once / 24h
},
```

Auth.js database sessions already extend `expires` on every `updateAge`-spaced request. No other code change needed. The behavior:
- A session's `expires` is first set to `now + 14d` at login.
- On any request ≥ 24h after the last `expires` update, the adapter rewrites `expires = now + 14d`.
- If the user is gone > 14d, session is deleted, user must re-login (magic link).

### 2.5 Landing page — restructure `app/page.tsx`

Five sections in a single RSC:
1. **Hero.** Brand + one-line value prop + two CTAs ("免费开始" → `/login`; "看模板" → anchor to #templates).
2. **Features.** 3 cards: 结构化分区 / 双模板 / 分享链接. Icon + title + 1-line copy each.
3. **Templates.** Two mini-previews rendering `<ClassicLayout content={demoResume} />` and `<ModernLayout content={demoResume} />` inside `transform scale-[0.38] origin-top-left` wrappers. Flip between them with shadcn `<Tabs>` (desktop) or stack vertically (mobile).
4. **FAQ.** `<Accordion>` with 4 Q&A (data safety, templates, pricing-free, export format).
5. **Footer.** Version tag, GitHub link, a simple terms link (reuses an empty stub `/terms`).

`lib/demo-resume.ts` exports a `demoResume: ResumeContent` fixture used only on Landing, ~50 lines of realistic bilingual sample data.

### 2.6 Mobile editor — Tabs switch

In `editor-client.tsx`: the current `grid lg:grid-cols-2` stays for lg. Below lg, render a shadcn `<Tabs value={view} onValueChange={setView}>` with two `<TabsContent>`: "编辑" (the left column's content) and "预览" (the right column's content). State is a single `view: "edit" | "preview"` useState, default "edit".

The toolbar (title input, template picker, share toggle, PDF download) stays visible across both tabs — it's above the tabs, so users can switch templates without leaving preview.

On lg, `<Tabs>` is bypassed via a `hidden lg:grid` vs `grid lg:hidden` twin that renders the same components twice. Acceptable duplication to keep both layouts pure — don't try to conditionally re-parent; that breaks react-hook-form state.

### 2.7 `metadata` cleanup

`app/layout.tsx`:
```ts
export const metadata = {
  title: { default: "intro-builder", template: "%s · intro-builder" },
  description: "面向互联网求职者的在线简历排版工具：结构化编辑、自动保存、一键导出 PDF。",
};
```
Per-page overrides on `/login`, `/dashboard`, `/resume/[id]/edit` via `export const metadata = { title: "登录" }` etc.

## 3. Responsive strategy

**Breakpoints used.**
- `sm`: 640 — ignore (no distinct layout)
- `md`: 768 — header collapses hamburger → inline nav; dashboard cards go 1 → 2 cols
- `lg`: 1024 — editor goes Tabs → side-by-side; dashboard 2 → 3 cols
- No `xl` overrides needed at this stage.

**Touch targets.** All interactive elements ≥ 44×44 pt on mobile. Re-check all `size="sm"` buttons in the editor toolbar — they may need to be `size="default"` below md.

**Typography.** Landing hero: `text-4xl md:text-5xl lg:text-6xl`. Section titles: `text-2xl md:text-3xl`. Everything else inherits Geist default.

## 4. Data flow (auth)

```
Request → middleware (proxy.ts) → auth() reads session cookie → session row in Postgres
                                                               ↓
                                                         extends expires
                                                         if updateAge elapsed
                                                               ↓
Protected page RSC → requireUserId() or auth() → renders with session
                                                               ↓
Header RSC → auth() → renders email / login button accordingly
```

No JWT, no refresh-token. Session is a cookie pointing to a DB row; adapter handles renewal.

## 7. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| `auth()` in Header RSC on every page = extra DB hit on every nav | Auth.js caches session within a request; one hit per page load is acceptable at 100 users/month |
| `scale-[0.38]` shrunk Layout in Landing renders at wrong aspect on phones | Wrap in fixed-aspect container; clip overflow; verified locally before ship |
| Tab-based mobile editor loses form state on switch | State lives in `react-hook-form`'s context provider which mounts once at the root of `EditorClient`; Tabs swap only render trees, not form state |
| Renaming `middleware.ts` → `proxy.ts` breaks Next 16's route matching | Trivial file rename, same export signature; confirmed by Next 16 deprecation note |
| shadcn `<Sheet>` in Header conflicts with `<Sheet>` on mobile editor preview tab | Different mount points, different state; no actual conflict |
